\chapter{Introduction}

\section{Purpose}
The purpose of this document is to provide a comprehensive description of the \textbf{SmartwareERP} Warehouse Management System (WMS). It specifies the functional and non-functional requirements, system architecture, external interfaces, and data models. SmartwareERP is a full-stack web application designed to digitize and streamline end-to-end warehouse operations including inbound receiving, AI-powered hazard classification, smart putaway, inventory tracking, order fulfilment (picking, packing, shipping), dispatch management, and real-time warehouse layout visualization.

\section{Document Conventions}
\begin{itemize}
    \item All mandatory requirements use the word ``shall''.
    \item Functional Requirements use IDs: FR-SWE-XXX.
    \item Non-Functional Requirements use IDs: NFR-SWE-XXX.
    \item Priority levels: High / Medium / Low.
\end{itemize}

\section{Intended Audience}
\begin{itemize}
    \item \textbf{Developers:} To guide the implementation and integration of modules.
    \item \textbf{Project Guides/Stakeholders:} To review the scope, architecture, and progress of the project.
    \item \textbf{Warehouse Operators (Admin/Staff):} To understand the capabilities and workflows of the system.
\end{itemize}

\section{Product Scope}
SmartwareERP is a web-based Warehouse Management System that provides:
\begin{itemize}
    \item \textbf{Inbound Receiving:} Receive goods into the warehouse, assign license plates, and record lot/serial/expiry information.
    \item \textbf{AI-Powered Hazard Classification:} Automatically classify products as INFLAMMABLE, TOXIC, FRAGILE, or NORMAL using the Google Gemini API.
    \item \textbf{Smart Putaway:} AI-driven rack suggestions based on hazard classification, zone compatibility, and rack capacity.
    \item \textbf{Inventory Management:} Real-time tracking of on-hand, allocated, hold, and damaged quantities with license-plate-level granularity.
    \item \textbf{Order Fulfilment:} End-to-end picking, packing, and shipping workflows with order tracking.
    \item \textbf{Dispatch Management:} License-plate-verified outbound dispatch with automatic rack load adjustments.
    \item \textbf{Warehouse Visualization:} Interactive warehouse layout and seat-map views showing zone utilization and rack-level inventory.
    \item \textbf{Alerts \& Reporting:} Low-stock alerts, expiry notifications, throughput analytics, and dock-to-stock time tracking.
    \item \textbf{Role-Based Access Control:} Separate Admin and Staff roles with permission-gated module access.
\end{itemize}

\chapter{Overall Description}

\section{Product Perspective}
SmartwareERP is a monolithic web-based application utilizing a Client-Server architecture. The frontend is built with React (via Vite) using Radix UI primitives and Tailwind CSS for styling. The backend is an Express.js server that exposes RESTful APIs, with data persisted in a PostgreSQL database through the Prisma ORM. AI-powered hazard classification is performed via the Google Gemini generative AI API.

\section{Product Functions}
High-level list of modules:
\begin{itemize}
    \item \textbf{Authentication Module:} Role-based login for Admin and Staff users with session persistence via localStorage.
    \item \textbf{Operations Dashboard:} Real-time KPIs including inbound pending count, orders to pick, orders to ship today, open exceptions, expiry alerts, hourly throughput charts, dock-to-stock time trends, active task queues, and replenishment alerts.
    \item \textbf{Product Master:} Full CRUD management of product catalogue with SKU, EAN, category, pricing, minimum stock levels, supplier linkage, and tracking type configuration.
    \item \textbf{Inbound Receiving:} Goods receipt with license plate assignment, lot/serial number tracking, expiry date recording, and AI-driven hazard classification on receive.
    \item \textbf{Smart Putaway:} AI-powered rack suggestion engine that matches product hazard classification to compatible zone-typed racks with capacity validation.
    \item \textbf{Rack Management:} CRUD for storage racks with zone type, capacity tracking, and real-time current load monitoring.
    \item \textbf{Warehouse Layout \& Seat Map:} Visual representation of warehouse zones (Receiving, Storage A/B/C, Packing, Dispatch) with rack utilization percentages, bin-level occupancy visualization, and drill-down item details.
    \item \textbf{Inventory Management:} Comprehensive inventory view with on-hand quantity, storage status (RECEIVED/STORED/SHIPPED), hazard classification badges, and search/filter capabilities.
    \item \textbf{Picking:} Order-driven picking workflow to prepare items for outbound fulfilment.
    \item \textbf{Packing:} Consolidation and packing of picked items for shipment.
    \item \textbf{Shipping:} Outbound dispatch with license plate verification and real-time inventory deduction.
    \item \textbf{Returns:} Module for handling returned goods (placeholder for future expansion).
    \item \textbf{Cycle Count:} Periodic inventory auditing module (placeholder for future expansion).
    \item \textbf{Reports \& Analytics:} Data-driven reports with charting capabilities using Recharts.
    \item \textbf{Settings:} System configuration and preference management.
    \item \textbf{Supplier Management:} Full CRUD for supplier records with contact details.
    \item \textbf{Customer Management:} Full CRUD for customer records with contact details.
    \item \textbf{Order Management:} Full CRUD for orders with order items, customer linkage, status tracking, and type classification (inbound/outbound).
    \item \textbf{Location Management:} CRUD for warehouse locations with zone, aisle, rack, shelf, capacity, and blocking support.
\end{itemize}

\section{Operating Environment}
\begin{itemize}
    \item \textbf{Operating System:} Windows / Linux / macOS (server-side), modern browsers (client-side)
    \item \textbf{Runtime:} Node.js
    \item \textbf{Frontend Framework:} React 18 with Vite 6
    \item \textbf{Backend Framework:} Express.js 5
    \item \textbf{ORM:} Prisma 7.x
    \item \textbf{Database:} PostgreSQL 14+
    \item \textbf{AI Service:} Google Gemini API (gemini-2.5-flash model)
    \item \textbf{Protocol:} HTTP/HTTPS (REST APIs on port 5000, frontend dev server on port 5173)
\end{itemize}

\section{Design and Implementation Constraints}
\begin{itemize}
    \item \textbf{Concurrency:} Database transactions (Prisma \$transaction) are used for critical multi-step operations like putaway and dispatch to prevent race conditions.
    \item \textbf{Zone Safety:} Putaway operations enforce zone-type compatibility -- a product classified as INFLAMMABLE can only be placed in a rack zoned for INFLAMMABLE storage.
    \item \textbf{Status Workflow:} Inventory follows a strict status lifecycle: RECEIVED $\rightarrow$ STORED $\rightarrow$ SHIPPED. Putaway is only allowed on RECEIVED items; dispatch is only allowed on STORED items.
    \item \textbf{AI Dependency:} Hazard classification requires a valid Google Gemini API key; the system gracefully degrades to ``NORMAL'' classification if the AI service is unavailable.
    \item \textbf{License Plate Tracking:} The serial number field doubles as a license plate identifier for inventory tracking throughout the warehouse lifecycle.
\end{itemize}

\section{Assumptions and Dependencies}
\begin{itemize}
    \item \textbf{Connectivity:} Requires network access for the frontend to communicate with the backend API and for the backend to reach the Gemini AI service.
    \item \textbf{Database:} A running PostgreSQL instance with the schema migrated via Prisma is assumed.
    \item \textbf{Third-Party Libraries:} Radix UI components, Tailwind CSS, Recharts, Lucide React icons, and other npm packages as specified in package.json.
    \item \textbf{Authentication:} The current version uses a hardcoded credential list for demonstration purposes; production deployment would require a proper authentication backend.
\end{itemize}

    \item \textbf{Picking/Packing/Shipping Pages:} Order-driven fulfilment workflow interfaces.

\textbf{Description:} The system shall enforce role-based access control. Admin users shall access all 14 modules. Staff users shall access only: Inbound, Putaway, Inventory, and Shipping.\\

\textbf{Outputs:} Filtered navigation menu and page access\\

\section{Inbound Receiving}

\textbf{Description:} Users shall be able to receive goods into the warehouse by specifying a product, quantity, receiving location, and optional lot number, serial number (license plate), and expiry date. The system shall create or update an inventory record accordingly.\\

\chapter{Data Requirements}

\section{Database Design}
The PostgreSQL database follows a relational schema managed by Prisma ORM with the following entity relationships:
\begin{itemize}
    \item Product (1:N) Inventory
    \item Product (1:N) OrderItem
    \item Product (1:N) InventoryTransaction
    \item Supplier (1:N) Product
    \item Customer (1:N) Order
    \item Order (1:N) OrderItem
    \item Location (1:N) Inventory
    \item Location (1:N) InventoryTransaction
    \item Inventory has a composite unique constraint on (productId, locationId, lotNumber, serialNumber)
\end{itemize}

\section{Glossary}
\begin{itemize}
    \item \textbf{License Plate (LP):} A unique identifier assigned to a unit of inventory for tracking through the warehouse lifecycle. Stored as the serial number field.
    \item \textbf{Putaway:} The process of moving received goods from the receiving zone to an appropriate storage rack.
    \item \textbf{Hazard Classification:} An AI-assigned category (INFLAMMABLE, TOXIC, FRAGILE, NORMAL) that governs safe storage placement.
    \item \textbf{Dock-to-Stock Time:} The elapsed time between receiving goods at the dock and completing putaway to a storage location.
    \item \textbf{Rack Zone Type:} A designation on storage racks that restricts them to hold only products with a matching hazard classification.
    \item \textbf{Seat Map:} A bin-level visualization of rack positions showing occupancy status and stored item details.
    \item \textbf{SKU:} Stock Keeping Unit -- a unique code identifying a product in the catalogue.
    \item \textbf{EAN:} European Article Number -- a standardized barcode symbology for product identification.
    \item \textbf{WMS:} Warehouse Management System.
    \item \textbf{ERP:} Enterprise Resource Planning.
    \item \textbf{ORM:} Object-Relational Mapping -- a programming technique for database abstraction (Prisma in this project).
\end{itemize}
